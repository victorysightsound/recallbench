## What Was Tried and Reverted

| Approach | Result | Why |
|----------|--------|-----|
| 5x over-fetch | No change | RRF already gets enough candidates at 3x |
| Decay session boost | -1.5% Recall@10 | Lets dominant sessions creep back in |
| Word-pair queries | -4.9% temporal | Adds noise, too many irrelevant matches |
| Topic-noun extraction | -1% multi-session | Single words too broad, adds API latency |
| Reranker (bigram) | -0.2% | Reshuffles good RRF ranking |

## Architecture

- **Embedding model:** all-MiniLM-L6-v2 (384-dim, BERT, local + API interchangeable)
- **Backends:** CandleNativeBackend (local), ApiBackend (DeepInfra), FallbackBackend (API→local)
- **Search:** FTS5 OR-mode + vector brute-force + RRF fusion
- **Diversification:** Max 1 chunk per session in multi-query search
- **Context assembly:** Multi-query (2 variants) → merge → diversify → budget fill

## Production Roadmap (Not Yet Implemented)

1. **ANN indexing** — Replace brute-force vector scan for >100K scale. sqlite-vec or HNSW.
2. **Recency weighting** — Boost newer memories in ranking. Test after MemoryAgentBench Conflict Resolution.
3. **Graph-enhanced search** — Traverse relationship edges after initial search. Requires LLM-assisted relationship detection during ingest. Tables exist, search integration pending.
