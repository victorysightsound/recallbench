# RecallBench + MindCore Status — 2026-03-24

## Major Pivot: Production Pipeline First

Stopped benchmark-focused optimization. Building the complete production
pipeline before further testing. Benchmarks tested mindcore in an artificial
way (chunking raw text without LLM extraction) that doesn't match production
usage. Production flow: LLM extracts individual facts → each fact is one
memory → graph edges between related/superseded facts → hybrid search.

## What's Built and Working

### MindCore
- **Hybrid search:** FTS5 OR-mode + vector (MiniLM-L6-v2) + RRF fusion
- **Embedding backends:** CandleNativeBackend (local), ApiBackend (DeepInfra), FallbackBackend
- **Graph infrastructure:** memory_relations table, GraphMemory CRUD, recursive traversal
- **Fact extraction:** Regex parser for structured fact lists (98.7% on MAB)
- **AssemblyConfig:** Configurable diversification, recency, graph depth, search limit
- **Feature flags:** vector-search, api-embeddings, local-embeddings, graph-memory, etc.

### RecallBench
- **Retrieval test harness:** Zero-cost search quality measurement, per-question JSON results
- **Embedding cache:** Pre-computed per dataset/variant/chunk_size/model
- **Session cache:** Raw dataset storage without chunking
- **Multiple system configs:** mindcore, mindcore-api, mindcore-mab, memloft
- **Datasets cached:** LongMemEval-S, ConvoMem (6 categories), MemoryAgentBench (4 splits)

## Optimal Search Configuration (for chunk-based retrieval)
- Chunk size: 1000 chars (tested 500/1000/2000)
- RRF k-values: (100, 20) for questions — vector-heavy
- Diversification: max 1/session (tested 1/2/3/unlimited)
- Multi-query: original + key-phrases (2 variants)
- Reranker: disabled (RRF ranking better alone)
- Embedding: all-MiniLM-L6-v2 (384-dim, local + API interchangeable)

## Key Learnings

1. **Graph needs fact-level granularity.** Chunk-level graph edges mix outdated and current facts in the same block, making both expansion and filtering counterproductive.

2. **Benchmarks don't test the real pipeline.** Production uses LLM extraction → individual facts. Benchmarks use chunking → block search. Different systems, different optimization targets.

3. **Diversification is dataset-dependent.** LongMemEval needs max 1/session (spread across sessions). MAB needs unlimited (focus on one session). Must be configurable.

4. **Recency weighting works for single-hop contradictions** but doesn't help multi-hop (need query decomposition or graph traversal at fact level).

5. **The retrieval test harness saved massive time and cost** by enabling parameter tuning without LLM tokens.

## ANN Design
Three modes: 'ann' (default, always on), 'exact' (brute-force for validation), 'off' (FTS5 only).
Not scale-dependent — faster at every scale, defaults to on.

## LLM Extraction Design
- Option A chosen: mindcore crafts prompts, calls LLM via callback trait, parses response
- Consumer provides LLM transport (any OpenAI-compatible API or CLI)
- Feature-gated: works without LLM (falls back to passthrough/chunking)
- Haiku recommended for extraction (10x cheaper than Sonnet, handles structured extraction well)
- Sonnet for implicit fact detection if needed (configurable by consumer)
